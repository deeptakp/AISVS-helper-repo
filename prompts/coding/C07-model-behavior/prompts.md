### C7-CODE-02: Implement Output Safety Classifier and Leak Prevention

**Objective**
- You are implementing output safety controls for an LLM application.

**Required Controls**
- Apply AISVS C7.3.1, C7.3.2, and C7.3.3.

**Execution Steps**
- Apply an automated content safety classifier to every model response before returning it to the user.
- Categories: violence, self-harm, hate, sexual content, dangerous instructions.
- Block and replace responses that exceed category thresholds.
- Implement a system prompt leak detector:
- Scan outputs for patterns that suggest disclosure of system prompt content, API keys, internal URLs, or backend data.
- Block outputs that match leak detection patterns.
- Prevent model-generated output from triggering outbound HTTP requests:
- Strip or neutralize URLs in generated outputs before they can be rendered by downstream systems.
- Implement a URL allow-list for any outputs that are permitted to contain links.
- Log all blocked outputs with: classifier score, block category, output hash, timestamp.
- Output: OutputSafetyLayer with content classifier, leak detector, and URL sanitizer.

**Expected Artifacts**
- OutputSafetyLayer with content classifier, leak detector, and URL sanitizer.

**Pass/Fail Rule**
- Mark PASS only when all required controls in scope are satisfied with evidence; otherwise mark FAIL and provide remediation.

### C7-CODE-03: Implement RAG Source Attribution

**Objective**
- You are implementing source attribution for a RAG-based AI application.

**Required Controls**
- Apply AISVS C7.4.1, C7.4.2, and C7.4.3.

**Execution Steps**
- Every RAG response must include citations linking claims to the retrieved source chunks.
- Attribution must be derived from retrieval metadata (document ID, chunk ID, source URL, retrieval timestamp)
- — it must NOT be generated by the model (the model cannot fabricate citations).
- Implement a citation integrity validator that:
- Verifies each cited chunk actually exists in the retrieved context passed to the model.
- Verifies each factual claim in the response can be traced to a retrieved chunk.
- Flags and surfaces unverifiable claims to the user with a "unverified" indicator.
- Include the retrieval metadata in the response object — do not discard it after inference.
- Output: RAG response formatter with metadata-derived attribution + citation integrity validator.

**Expected Artifacts**
- RAG response formatter with metadata-derived attribution + citation integrity validator.

**Pass/Fail Rule**
- Mark PASS only when all required controls in scope are satisfied with evidence; otherwise mark FAIL and provide remediation.

**Source Prompt**

Apply AISVS C7.4.1, C7.4.2, and C7.4.3.

Requirements:

### C7-CODE-04: Implement Confidence-Based Fallback

**Objective**
- You are implementing reliability controls for an LLM application.

**Required Controls**
- Apply AISVS C7.2.1 and C7.2.2.

**Execution Steps**
- Implement a confidence estimation step for every model response:
- Options: model self-reported logit entropy, ensemble consistency score, or external verifier.
- Normalize confidence to a 0.0–1.0 scale.
- Define a configurable confidence threshold (default: 0.7).
- If confidence drops below the threshold:
- Block the response and return a configured fallback message ("I'm not confident enough to answer this reliably.").
- Log the low-confidence event with the estimated confidence score and prompt hash.
- (L3) For responses classified as high-risk by policy: route to an additional verification step before returning.
- Output: ConfidenceEstimator + threshold-based fallback handler + high-risk routing.

**Expected Artifacts**
- ConfidenceEstimator + threshold-based fallback handler + high-risk routing.
